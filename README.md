## 협업 규칙 (필독!)

과제 평가 항목인 **개별 커밋 수**를 공정하게 기록하고, 모든 팀원이 깃허브 웹사이트 환경에서 안전하게 협업하기 위해 다음 규칙을 준수합니다.

### 1. 기본 작업 방식: 웹에서 직접 수정

별도의 프로그램(Git CLI, Github Desktop 등)을 사용할 필요는 없습니다.
모든 작업은 **깃허브 웹사이트**에서 이루어집니다.

1.  수정할 파일(`RULES.md`)로 이동합니다.
2.  오른쪽 상단의 **연필 모양(Edit this file) 아이콘**을 클릭합니다.
3.  내용을 수정합니다.
4.  페이지 하단의 **'Commit changes'** 녹색 버튼을 눌러 작업을 저장합니다.

이후에 개정 단계에선 조항을 수정한 뒤, **반드시 파일 하단의 `5. 개정 내역 정리표`에 본인의 수정 사항을 한 줄 추가**합니다.

---

### 2. 커밋 규칙

1.  **자주 커밋하기**: 작업한 내용이 사라지지 않도록 **틈틈이 자주 커밋(저장)하는 것을 권장합니다.**
2.  **구체적으로 기록하기**: 커밋(저장) 시, **어떤 작업을 했는지 구체적으로 기록**해야 합니다.
    * 기본 메시지(예: `Update RULES.md`)를 그대로 사용하지 마시고, 본인의 작업 내역을 다른 팀원이 알아볼 수 있게 명확히 적어주세요.

**좋은 예시:**
* `제3조 소음 관리 조항 신설 및 개정표 반영`
* `1조 오탈자 수정 및 벌칙 문구 구체화`
* `README 팀원 역할 분담표 작성`

---

### 3. 충돌 발생 시 대처 방법

작업을 저장(Commit)하려 할 때, 다른 팀원이 방금 나와 같은 부분을 수정했다면 충돌(Conflict)이 발생합니다.

이때는 `main` 브랜치로 바로 저장이 불가능하며, 아래와 같은 화면이 나타납니다.

1.  커밋 창 상단에 `There was an error committing your changes:...`라는 경고 문구가 생깁니다.
2.  `Commit directly to the main branch` 말고 **`Create a new branch for this commit and start a pull request.`** 옵션을 선택합니다. (이것만 선택될 것입니다.)
3.  아래쪽 **'Propose changes'** 녹색 버튼을 클릭합니다.
4.  (자동으로 'Pull Request' 페이지가 열립니다.) 그냥 **'Create pull request'** 버튼을 한 번 더 누릅니다.
5.  **그 다음에 저(임민규)에게 알려주시면 제가 확인 후 해결하겠습니다.**

### 4. 피어리뷰 및 개정 절차 (개정용 파일 사용)

복잡한 Pull Request 방식 대신, `개정.md`라는 별도의 개정용 파일을 사용합니다. 이를 통해 누구나 충돌 걱정 없이 자유롭게 수정안을 제안하고 피어리뷰를 진행할 수 있습니다.

**A. 개정용 파일(`개정.md`)에 수정안 제안하기**

1.  개정 기간에는 원본인 `RULES.md`가 아닌, **`개정.md` 파일을 수정**합니다.
2.  수정하고 싶은 조항 바로 아래에, `>` 기호를 이용해 **인용 블록(blockquote)을 추가**하여 제안 내용을 작성합니다. **기존 조항은 절대 수정하지 마세요.**
3.  인용 블록 안에는 다음 내용을 포함합니다.
    *   **수정 제안 제목**: `> **[제7조 2항 수정 제안]**` 과 같이 어떤 조항을 수정하는지 명확히 적습니다.
    *   **제안 내용**: 수정하려는 조항의 전체 내용을 다시 작성합니다. 이때 **추가할 내용은 `**굵게**`**, **삭제할 내용은 `~~취소선~~`**으로 표시합니다.
    *   **제안 이유 및 의견**: `> **(의견/이름):** 제안 이유나 의견을 자유롭게 작성합니다.` 형식으로 작성합니다.
4.  아래 예시를 참고하여 작성해 주세요. (`개정.md` 파일 상단의 템플릿을 복사해서 사용해도 좋습니다.)
    ```markdown
    2. 외부인(공사 인력, 외부 업체 직원 등)의 교내 출입에 대해... (기존 조항)
    > **[제7조 2항 수정 제안]**
    >
    > **제안 내용:** 2. 외부인(공사 인력, 외부 업체 직원, **배달원** 등)의 교내 출입에 대해...
    >
    > **(의견/임민규):** 최근 배달 오토바이의 교내 난폭 운전 및 흡연 문제가 잦아...
    ```
5.  작성이 끝나면 페이지 하단의 **'Commit changes'**에서 변경 내용을 명확히 요약하여 적고 저장합니다. (예: `[제안] 제7조 2항 수정`)

**B. 제안된 내용에 대해 의견 남기기 (피어리뷰)**

1.  팀원들은 `개정.md` 파일에서 다른 사람이 제안한 수정안을 확인합니다.
2.  의견을 추가하고 싶다면, 해당 제안의 **인용 블록(`>`) 안으로 들어가** `> **(의견/이름):** 코멘트` 형식으로 자신의 의견을 추가하고 커밋합니다.
    *   **예시:**
        ```markdown
        > **[제7조 2항 수정 제안]**
        > ...
        > **(의견/임민규):** ...
        > **(의견/김재욱):** 좋은 의견입니다. 저도 찬성합니다.
        ```
3.  찬성, 반대, 또는 추가 수정 제안 등 자유롭게 의견을 남길 수 있으며, 이 과정 역시 개인의 커밋으로 기록됩니다.

**C. 합의된 내용을 원본(`RULES.md`)에 최종 반영하기**

1.  개정 기간이 끝나면, 조장이 `개정.md`에 제안된 수정안들과 의견들을 종합적으로 검토합니다.
2.  팀원들과의 논의를 통해 **최종 합의된 내용**을 복사하여 원본 **`RULES.md` 파일에 적용**합니다.
3.  모든 내용이 원본에 반영되면, 다음 개정을 위해 `개정.md` 파일의 내용을 다시 원본 `RULES.md`와 동일하게 초기화합니다.
